swarm-loop-review

local + GitHub PR review · design one-pager · human interface & agent flow · new skill, separate from /rpr

One command runs a multi-agent review of a PR (or a local diff), resolves most disagreements itself, and posts a single review to GitHub — surfacing only the genuine judgment calls for you.

Prerequisite: the repo must ship a review-double-checks.md — its codebase-specific review standards. The skill carries no project rules itself; type-safety, fail-fast, naming, etc. live there.

How you run it

swarm-loop-review [target] [mode] [flags]

Target — auto-detected if omitted (first match wins):

  • <pr-url | number> → that PR
  • else current branch has an open PR → review that PR
  • else uncommitted changes (e.g. on main) → local diff
  • else branch ahead of base, no PR → local diff vs base
  • else → nothing to review

PR exists and you have uncommitted edits → PR wins; pass --diff to force local.

Mode — what it does with the code:

  • collaborate (default) — review, synthesize, recommend. Does not touch code. You decide what to act on.
  • fix — applies the net-positive changes itself, re-reviews until clean, then posts.

Flags:

  • mini — faster: just two reviewers (one all-areas Claude + Codex), a single debate round. For quick passes.
  • local — print the review to the conversation instead of posting to GitHub (automatic when there is no PR).
  • base <branch> — set the diff base (default: the PR base / main).

Combine freely: swarm-loop-review fix mini   swarm-loop-review 1234 base develop

How it decides what to fix

No blanket default across all findings. The triage agent uses its own judgment, but defaults to fixing the two areas it habitually under-acts on:

  • violations of the repo’s review-double-checks.md conventions (for minerva: type-safety, fail-fast / fail-loud, …)  ·  over-explaining / change-narrating comments

Nits are fixed only when clearly net-positive — never “fixed” if the change would make things worse.

What it produces

Every kept finding is debated (kill ⇄ rescue) until both sides stop moving. The result sorts into three buckets:

  • Agreed → posted as actionable comments, each tagged issue suggestion nit with per-section IDs (C1, S1…).
  • Contested → posted flagged “reviewer flagged, author dismissed — your call.” This is the only thing that needs you.
  • Dismissed → dropped silently, and remembered so re-runs don’t re-raise them.

What you see

  • On GitHub — one review (comment-only, never request-changes / approve), with a header noting it is automated. Re-runs supersede rather than pile up.
  • In fix mode — a changelog of what was changed / dismissed / left contested.
  • Locally — the same review, printed in the conversation.

Your job: arbitrate the Contested bucket. Everything else is handled.
